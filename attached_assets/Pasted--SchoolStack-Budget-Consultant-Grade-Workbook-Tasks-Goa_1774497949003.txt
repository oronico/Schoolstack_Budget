## TASK 1: Input cell formatting (blue = editable)

**Priority:** Highest. This is the single most recognizable signal that a workbook is professionally built. Every consultant model uses this convention.

### The convention

- **Blue (or light blue) background + blue font** = editable input assumption
- **White/no background + black font** = formula-driven output (do not touch)
- **Light gray background** = label/header row

### Implementation

#### Step A: Define shared styles in `workbook-helpers.ts`

Add these constants near the top of the file (alongside existing formatting constants):

```typescript
// Consultant-grade cell formatting conventions
export const INPUT_CELL_FILL: ExcelJS.Fill = {
  type: "pattern",
  pattern: "solid",
  fgColor: { argb: "FFDCE6F1" },  // light steel blue
};

export const INPUT_CELL_FONT: Partial<ExcelJS.Font> = {
  color: { argb: "FF1F4E79" },  // dark navy blue
  bold: false,
};

export const INPUT_CELL_BORDER: Partial<ExcelJS.Borders> = {
  bottom: { style: "thin", color: { argb: "FF8DB4E2" } },  // subtle blue underline
};

// Combined style object for easy application
export const INPUT_STYLE = {
  fill: INPUT_CELL_FILL,
  font: INPUT_CELL_FONT,
  border: INPUT_CELL_BORDER,
};

export const FORMULA_CELL_FONT: Partial<ExcelJS.Font> = {
  color: { argb: "FF000000" },  // black
};
```

#### Step B: Apply to the Assumptions tab on ALL export formats

The Assumptions tab exists on every export format. Every cell that contains a user-entered value (enrollment counts, tuition rates, salary amounts, escalation percentages, etc.) should get `INPUT_STYLE`. Every cell that contains a calculated value should get `FORMULA_CELL_FONT`.

In each export file, find where assumption values are written to cells. For example, in the lender pro forma:

```typescript
// BEFORE
cell.value = enrollmentYear1;

// AFTER
cell.value = enrollmentYear1;
cell.fill = INPUT_CELL_FILL;
cell.font = INPUT_CELL_FONT;
cell.border = INPUT_CELL_BORDER;
```

Or create a helper function:

```typescript
function applyInputStyle(cell: ExcelJS.Cell) {
  cell.fill = INPUT_CELL_FILL;
  cell.font = INPUT_CELL_FONT;
  cell.border = INPUT_CELL_BORDER;
}

// Usage
const cell = ws.getCell("B5");
cell.value = enrollmentYear1;
applyInputStyle(cell);
```

#### Step C: Apply to other tabs where assumptions are referenced

On the 5-Year Model, Drivers, and forecast tabs, cells that REFERENCE the Assumptions tab via formula should remain black (formula output). But any direct-entry values on those tabs (if any exist) should also get input styling.

#### Step D: Add a legend row

On the Assumptions tab (all formats), add a legend row near the top:

```
Row 2: [Blue cell] = "Editable assumption - change this value"    [White cell] = "Calculated - driven by formula"
```

This takes 2 cells and instantly communicates the convention to anyone opening the file.

### What this looks like

Open the workbook. Every blue cell is something you can change. Every black cell recalculates automatically. A lender looking at the model can immediately see “if I change enrollment from 100 to 80, what happens to the P&L?” That’s the consultant experience.

### Verify

- Open each export format in Excel
- Confirm assumption values are blue-on-light-blue
- Confirm formula cells are black-on-white
- Confirm the legend row appears on the Assumptions tab
- Change a blue cell value and confirm formulas recalculate
- Run `qa:golden` and `qa:excel` – all pass

## TASK 2: Instructions tab on all export formats

**Priority:** High. A workbook without instructions is a spreadsheet. A workbook with instructions is a deliverable.

### What to build

Add a new “Instructions” tab as Tab 1 (first tab) on ALL export formats. The 21-tab model already has one – use that as the template but make it work across all formats.

### Content for the Instructions tab

```
ROW 1: [SchoolStack Budget logo area]
       School Name: {schoolName}
       Prepared: {date}
       Model Type: {schoolType} | {schoolStage}

ROW 4: HOW TO USE THIS MODEL

ROW 6: 1. UNDERSTAND THE COLOR CODING
       Blue cells (shaded) are editable assumptions. Change these to test different scenarios.
       Black cells are formulas. They recalculate automatically. Do not overwrite them.

ROW 9: 2. START WITH ASSUMPTIONS
       Go to the Assumptions tab first. Review each input: enrollment, tuition rates,
       staffing, and expenses. These drive everything else in the model.

ROW 12: 3. REVIEW THE 5-YEAR PROJECTION
        The 5-Year Model tab shows your full P&L. Look for:
        - Does revenue grow faster than expenses?
        - Is there a surplus (positive net income) by Year 2-3?
        - Are facility costs under 15% of revenue?

ROW 17: 4. KEY RATIOS TO WATCH
        - Payroll as % of Revenue: Target under 65%. Over 80% is critical.
        - Operating Surplus Margin: Target over 5%. Negative means you're losing money.
        - DSCR (if you have debt): Target over 1.25x. Under 1.0x means you can't cover payments.
        - Revenue per Student: Varies by model, but under $5,000/student is very tight.

ROW 23: 5. WHAT TO DO NEXT
        - Adjust assumptions and watch the model recalculate
        - Share this workbook with your board, lender, or advisor
        - Visit budget.schoolstack.ai to update your model online
        - Questions? Email allison@schoolstack.ai

ROW 28: DISCLAIMER
        This model is for planning purposes. It does not constitute financial advice.
        Projections are based on the assumptions you entered and may not reflect actual results.
        SchoolStack is a product of Building Hope Impact Fund.
```

### Implementation

Create a shared function in `workbook-helpers.ts`:

```typescript
export function addInstructionsSheet(
  workbook: ExcelJS.Workbook,
  schoolName: string,
  schoolType: string,
  schoolStage: string,
  tabCount: string,  // e.g., "3-tab", "8-tab", "21-tab"
): ExcelJS.Worksheet {
  const ws = workbook.addWorksheet("Instructions", {
    properties: { tabColor: { argb: "FF328555" } },  // Evergreen
  });

  ws.getColumn(1).width = 4;   // gutter
  ws.getColumn(2).width = 80;  // content
  ws.getColumn(3).width = 30;  // secondary

  // ... write the content rows above using ws.getCell() ...
  // Use Quicksand-style bold for headings, Nunito for body
  // Apply SchoolStack brand colors

  ws.pageSetup = {
    orientation: "portrait",
    fitToPage: true,
    fitToWidth: 1,
    fitToHeight: 0,
    printArea: "A1:C35",
  };

  return ws;
}
```

Then call it from each export file, inserting the Instructions tab BEFORE the existing tabs:

```typescript
// In formula-export.ts, lender-proforma-export.ts, underwriting-workbook.ts
addInstructionsSheet(workbook, schoolName, schoolType, schoolStage, "3-tab");
// ... then add the other tabs as usual
```

**Lender Pro Forma (currently 8-tab, becomes 9):**

**21-tab model:**
Already has Instructions. Review it for consistency with the content above and add the color-coding legend if missing.

## TASK 3: Benchmark dashboard tab

**Priority:** High. This is what makes the workbook replace a consultant’s opinion. The numbers speak for themselves with green/yellow/red formatting.

### What to build

A new “Financial Health” tab (or “Dashboard”) that shows 7-8 key ratios with conditional formatting. Place it as the LAST tab so it feels like a summary/scorecard.

### Metrics to include

These come directly from the consultant engine thresholds (already defined in the codebase):

|Row|Metric                  |Formula source                        |Healthy (green)|Watch (yellow)|Critical (red)  |
|---|------------------------|--------------------------------------|---------------|--------------|----------------|
|1  |Revenue per Student     |Total Rev Y1 / Enrollment Y1          |> $8,000       |$5,000-$8,000 |< $5,000        |
|2  |Payroll % of Revenue    |Total Payroll Y1 / Total Rev Y1       |< 65%          |65-80%        |> 80%           |
|3  |Facility % of Revenue   |Facility Cost Y1 / Total Rev Y1       |< 15%          |15-25%        |> 25%           |
|4  |Operating Surplus Margin|Net Income Y1 / Total Rev Y1          |> 5%           |0-5%          |< 0%            |
|5  |Revenue Sources         |Count of non-zero categories          |3+             |2             |1               |
|6  |Cash Runway (months)    |Cash reserve / monthly burn           |> 3 months     |1-3 months    |< 1 month       |
|7  |DSCR (if debt)          |NOI / Debt Service                    |> 1.25x        |1.0-1.25x     |< 1.0x          |
|8  |Break-even Year         |First year with cumulative positive NI|Year 1-2       |Year 3-4      |Year 5+ or never|

## TASK 5: Un-flatten program-level revenue in the 21-tab export

**Priority:** Highest complexity, highest long-term value. This is the edit that makes the model genuinely flexible by program.

### The problem

The wizard captures per-program data:

- Program name (e.g., “K-5 General Ed”, “6-8 STEM Academy”)
- Enrollment per year per program
- Tuition rate per program with escalation
- Per-pupil funding per program
- Discount tiers per program (for private/tuition-based)

But the 21-tab export’s Enrollment Drivers and Tuition & Funding tabs blend everything into aggregated totals. A lender can’t see that Program A has 30 students at $12K tuition while Program B has 20 students at $8K ESA funding. They can’t stress-test “what if Program B enrollment drops 30%?”

### What it should look like

**Enrollment Drivers tab:**

```

6-8 STEM Academy           20        30        40        45        50    [blue cells]
TOTAL ENROLLMENT           50        70        90       100       110    [formula: SUM]
Max Capacity              120       120       120       120       120    [blue cell]
Utilization               42%       58%       75%       83%       92%    [formula]
```

**Tuition & Funding tab:**


### Implementation approach

The model data (`ModelFormData`) already has `revenueRows[]` with each row containing:

- `name` (the revenue source name)
- `category` (tuition_fees, public_funding, philanthropy, etc.)
- `driverType` (per_student, fixed, etc.)
- `perStudentAmount` or `fixedAmount`
- `escalationRate`
- `startYear`, `endYear`
- `programId` (links to a specific enrollment program)

The un-flattening requires:

1. **Group revenueRows by category and programId**
1. **For each program-linked row**, render enrollment ref + rate + escalation + subtotal as separate Excel rows with formulas
1. **For non-program rows** (fixed grants, philanthropy), render as simple rows
1. **SUM formulas** roll up subtotals to category totals to grand total

#### Key files to modify:

**`underwriting-workbook.ts`** – the `buildEnrollmentDriversTab()` and `buildTuitionFundingTab()` functions (or wherever these tabs are generated). Instead of writing a single blended enrollment row, iterate over `enrollment.programs[]` and write a row per program.

**`workbook-helpers.ts`** – may need helper functions to group revenue rows by program and compute per-program totals.

#### Critical: Maintain formula integrity

The 5-Year Operating Statement, Monthly Cash Flow, and Budget Summary tabs reference revenue totals. Currently they probably reference a single blended total cell. After un-flattening:

- Each program subtotal should be a SUM formula
- Category totals should SUM the subtotals
- Grand total should SUM the categories
- Downstream tabs should reference the grand total cell (same cell address if possible)

If the grand total cell address changes, update all cross-tab references.

### What about the 3-tab and 8-tab exports?

The 3-tab public export is intentionally simpler. Blended totals are fine there. The 8-tab lender pro forma should ideally show program-level breakdown on the Drivers tab if feasible, but this is lower priority than the 21-tab model.

**Recommended scope for alpha:** Un-flatten the 21-tab model only. The 3-tab and 8-tab can remain blended for now.
