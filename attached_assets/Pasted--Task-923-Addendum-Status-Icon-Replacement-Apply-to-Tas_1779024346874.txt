## Pinned decisions

These resolve the open choices in the base spec. Apply directly:

### Icon mapping (Option A confirmed)

```
[+]  →  ✓   (Healthy / Pass)
[~]  →  ⚠   (Watch closely)
[!]  →  ✕   (Critical / Needs attention)
[ ]  →      (no glyph — neutral header, plain label only)
```

Rationale: checkmarks and X’s carry semantic meaning at a glance, requiring no legend lookup. Option B (geometric shapes ●○◐) was considered and rejected because shapes without color require legend recall on every read.

If PDFKit font rendering prevents any of the four glyphs above after Task #922 lands, fall back per-glyph to:

```
✓  →  [OK]
⚠  →  [WATCH]
✕  →  [CRITICAL]
```

Document any per-glyph fallback in the helper function comments with the codepoint and rendering test result.

### Color scope

Stay monochrome for this task. Do NOT apply green/amber/red to the icons in #923.

File a follow-up task on close of #923:

```
Task: Status icon semantic color application
Scope: Apply green (✓), amber (⚠), red (✕) to status icons in 
  lender and board packet PDFs
Depends on: #923 (icon swap), #922 (font / color rendering capability)
Rationale: Monochrome icons read as flat; color is what makes 
  status communicate at a glance for lender review.
```

### Sibling bug search step

```
Step: Sweep for all bracketed-placeholder patterns beyond the named four.

Pattern: bracketed single-character placeholders in source templates
Search scope: artifacts/api-server/src/lib/packets/*.ts

Discovery command:
  grep -rohP "\[[+~!\*\?xo •\-]\]" --include="*.ts" \
    artifacts/api-server/src/lib/packets/ | sort -u > /tmp/placeholder_patterns.txt

Action:
  1. Run the discovery command. Output is a deduplicated list of every 
     bracketed-placeholder pattern in source.
  2. For each pattern in /tmp/placeholder_patterns.txt:
     a. Note where it's used (grep -rln for each pattern)
     b. Decide: route through renderStatusIcon, preserve as-is (e.g., 
        legitimate non-status bracket usage), or remove
     c. Document the decision in the helper function comments
  3. Update renderStatusIcon to handle any newly discovered status 
     patterns. Update regression test to assert the broader pattern set.

Expected patterns based on prior review: [+] [~] [!] [ ]
Possible additional patterns: [*] [?] [x] [o] [•] [-]
```
