# Task #930 — Pre-Launch Math Integrity & Primary Data Source Verification

**Type:** Pre-launch readiness gate
**Blocks:** Go-live for beta founders, Phase 2 funder demos, Lending Lab Cycle 2 packet review
**Estimated effort:** 2–3 days
**Pairs with:** Verification Protocol + Addendum (existing)

---

## What & Why

After the #909–#929 remediation sweep, the system has been hardened against a known set of drift patterns. Before going live with real founders, lenders, and funders, the math must be verified end-to-end against a designated primary data source, with explicit confirmation that the engine, the workbooks, and the PDFs all agree.

This task is a verification gate, not a refactor. It produces a math-integrity report covering every financial output the system produces, traced to its canonical source. Any discrepancy uncovered is either resolved in scope (if it's a routing bug) or filed as a blocker follow-up (if it's a calculation issue).

The primary data source designation is the architectural component: every downstream surface (engine intermediate, workbook tab, PDF section, in-app card, exported file) must trace back to one canonical source per metric. After this task, "where does this number come from?" has exactly one answer per metric.

---

## Done looks like

1. **Primary Data Source Registry** — A single document (in-code or in `docs/`) names the canonical source for every financial metric the system produces. Every metric appears exactly once with a designated source.

2. **Math integrity report** — For each persona fixture, every numeric figure in every output surface has been traced to its primary source and confirmed to match within rounding. A structured report (CSV or markdown) lists every metric, every surface where it appears, the value rendered, and the source-of-truth value.

3. **Zero unresolved discrepancies** — Any drift between sources and surfaces is either fixed in scope (routing bug) or filed as a blocker task that gates go-live.

4. **Cross-surface harness** — A test that runs the verification automatically. When a new persona fixture is added or an engine calculation changes, the harness re-runs and flags any new drift. This is the standing safeguard after this task closes.

5. **Go-live gate checklist** — A documented list of conditions that must be true before generating packets for real founders. The harness passing is one item; others include workbook download integrity, PDF render integrity at scale, and at least one human-reviewed end-to-end sample.

## Primary Data Source Registry

Build a registry covering every metric the system produces. Structure:

```typescript
type CanonicalMetric = {
  metricId: string;                    // e.g., 'y1_revenue', 'y1_dscr_normalized'
  description: string;
  primarySource: {
    location: string;                  // file:function or workbook!cell range
    type: 'engine_function' | 'workbook_cell' | 'derived';
    formula?: string;                  // human-readable if derived
  };
  consumers: ConsumerLocation[];       // every place this metric appears
  unit: 'USD' | 'PCT' | 'COUNT' | 'MONTHS' | 'RATIO';
  rounding: { precision: number; method: 'round' | 'truncate' | 'banker' };
  notes?: string;
};

type ConsumerLocation = {
  surface: 'engine' | 'workbook_5yr' | 'workbook_proforma' | 'lender_pdf' 
         | 'board_pdf' | 'in_app_card' | 'export_json';
  location: string;                    // section name, cell, component
  transformation?: string;             // any unit/format change applied
};
```

The registry covers, at minimum, the metrics listed below. The sibling search step identifies any additional ones.

## Math integrity verification procedure

For each persona fixture in the test suite (Riverside, Liberty, Oakwood, plus any additional fixtures):

### Step 1 — Render all outputs

Generate every output surface end-to-end:
- 5-Year Financial Model workbook (`.xlsx`)
- Lender Pro-Forma workbook if it ships (`.xlsx`)
- Lender packet PDF
- Board packet PDF
- In-app dashboard state (snapshot JSON)
- Any export endpoints (JSON, CSV)

### Step 2 — Extract every numeric value

For each surface, programmatically extract every numeric value rendered. Tools:

- **Workbook extraction:** Read every cell with a numeric value, including formula-derived cells (post-recalc). Tag with sheet name, cell reference, value.
- **PDF extraction:** Extract text, parse for numeric tokens with context (preceding label, surrounding paragraph). Use a regex catalog covering currency (`\$[\d,]+`), percentages (`[\d.]+%`), DSCR (`[\d.]+x`), counts (raw integers).
- **In-app state:** Serialize the rendered component state; extract all numeric props/fields.
- **Export endpoints:** Parse the response payload; extract all numeric fields.

### Step 3 — Map extracted values to registry metrics

For each extracted value, identify which `CanonicalMetric` it represents. This is the hardest part — text near the value provides context (e.g., "Year 1 DSCR: 21.01x" maps to `y1_dscr_normalized`). Build a label-to-metric mapping during this step; future runs reuse it.

Values that cannot be mapped to a registry metric are flagged. They are either:
- Metrics the registry missed (add to registry)
- Cosmetic numbers like page numbers (annotate and exclude)
- Bugs where the surface renders a number with no clear source (the highest-priority finding)

### Step 4 — Compute the primary source value

For each registry metric, compute the canonical value from the primary source. Workbook cells are read post-recalc. Engine functions are called with the persona's input set and the result captured. Derived metrics are computed from their dependencies using the documented formula.

### Step 5 — Compare and report

For each (metric, surface) pair, compare the extracted value to the canonical value. The comparison tolerance is per-metric:

- Currency: ±$1 (allows for rounding-display differences)
- Percentages: ±0.05 percentage points
- DSCR / ratios: ±0.01
- Counts: 0 (exact match required)
- Months: ±0.1

Any pair outside tolerance is a discrepancy. Report shape:

```
| metricId | surface | location | extracted | canonical | delta | severity |
```

### Step 6 — Triage discrepancies

For each discrepancy:

- **Routing bug** (surface reads wrong source): fix in scope. The fix routes the surface to the canonical primary source.
- **Calculation bug** (canonical source itself is wrong): file as a blocker follow-up. Do not paper over with a workaround.
- **Unit/format issue** (value is right, formatting is wrong): fix in scope.
- **Acceptable variance** (documented intentional difference, e.g., normalized vs as-planned): annotate in the registry; not a discrepancy.

The math integrity report includes the triage decision for every discrepancy.

## Cross-surface harness

After the verification report passes, the procedure becomes a standing test:

```typescript
// pseudocode
describe('Cross-surface math integrity', () => {
  for (const fixture of personaFixtures) {
    it(`${fixture.name}: every metric matches across all surfaces`, async () => {
      const outputs = await renderAllSurfaces(fixture);
      const extracted = extractAllValues(outputs);
      const canonical = computeCanonicalValues(fixture);
      
      const discrepancies = compareWithTolerances(extracted, canonical);
      
      expect(discrepancies).toEqual([]);
    });
  }
});
```

The harness runs on every CI build. New persona fixtures are added by dropping them into the fixture directory; no test code changes required. New metrics are added by extending the registry; the harness picks them up automatically.

## Steps

1. **Build the Primary Data Source Registry.** Start with the metrics listed above; extend via the sibling search step.

2. **Run the sibling search.** Identify every metric the system produces that isn't yet in the registry. Likely sources of missed metrics: in-app cards, exported JSON shapes, board packet sections, decision-comparison PDFs.

3. **Build the extraction tooling.** Workbook reader, PDF text-with-context extractor, in-app state serializer. These become reusable utilities for the standing harness.

4. **Build the canonical computation layer.** Function per metric that returns the primary source value for a given fixture.

5. **Run verification for all persona fixtures.** Produce the math integrity report.

6. **Triage every discrepancy.** Fix routing bugs in scope; file calculation bugs as blocker follow-ups.

7. **Wire the harness into CI.** Add to the test suite that runs on every build.

8. **Document the go-live gate checklist.** Make it visible to anyone with merge authority.

9. **Hand-review one sample end-to-end.** This is non-automatable. Allison, Lance, or a designated reviewer walks through a complete packet and signs off.

  # Find every dashboard card
  find artifacts/school-financial-model/src/components/ -name "*Card.tsx"

Action:
  1. List every metric that appears in any output surface.
  2. Cross-reference against the draft registry.

     c. Add the consumers to the entry

### Explicit regression assertion (subsystem-level, covering all patterns)

```
Assertion target: End-to-end math integrity across all output surfaces

Subsystem-level assertions:

  For every persona fixture:
    For every metric in the Primary Data Source Registry:
      Let canonical = computeCanonicalValue(fixture, metric)
      For every consumer in metric.consumers:
        Let extracted = extractValueAtConsumer(fixture, consumer)
        Assert: |extracted - canonical| <= metric.tolerance
      Assert: no consumer is missing from the registry's consumer list

  For the registry itself:
    Assert: every metric has at least one consumer
    Assert: no consumer location appears in two different metric entries 
      (no surface should render the same field via two different metrics)
    Assert: every metric has a documented primarySource location

  For the harness coverage:
    Assert: every numeric value extracted from any output surface 
      maps to a registry metric
    Assert: orphan values (extracted but not in registry) produce a 
      test failure listing the surface and value

Persona-specific expectations:
  For each fixture, the harness produces a zero-discrepancy report.

Failure reporting:
  "Math integrity discrepancy: {metric} in {fixture} at {surface}/
   {location} — canonical {canonical_value}, rendered {extracted_value}, 
   delta {delta}, tolerance {tolerance}"
```

## Prevention mechanism

The registry + harness combination prevents drift permanently:

1. **Registry as living document:** Every future task that introduces a metric updates the registry. Reviewers reject PRs that add metrics without registry entries.

2. **Harness in CI:** New consumers added without updating the registry produce test failures. Changes to canonical sources that don't propagate to consumers produce test failures.

3. **Orphan detection:** Numbers rendered without a registry entry are caught at the harness layer. The system cannot silently produce values without a documented source.

4. **Migration discipline for breaking changes:** Any change to a canonical source includes updates to all consumer locations in the same PR. The harness enforces this at merge time.
