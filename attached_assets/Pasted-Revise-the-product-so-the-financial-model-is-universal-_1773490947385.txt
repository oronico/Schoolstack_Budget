Revise the product so the financial model is universal across school types and uses accounting categories.

IMPORTANT PRODUCT CHANGE
Do not assume every school is tuition-based.
Do not assume every staff member is full-time.
Do not limit expenses to a simplified startup budget.
The model must work for:
- new schools
- operating schools
- private / tuition-based schools
- charter / public-funded schools
- hybrid / mixed-funding schools

Keep the experience simple and polished, but make the underlying structure robust and accounting-aligned.

NEW MODEL DIMENSIONS
Add two dimensions to every model:

1. SCHOOL STAGE
- new_school
- operating_school

2. FUNDING PROFILE
- tuition_based
- charter_public_funded
- hybrid_mixed

Use these to drive defaults, visible fields, consultant language, and export labels.
Do NOT fork the app into separate products.

UNIVERSAL MODEL RULE
Build one common financial engine with:
- configurable revenue streams
- FTE-based staffing
- accounting-category expense structure
- school-type-aware consultant guidance

UI PRINCIPLE
The app should still feel clean and founder-friendly.
Use plain-English labels in the UI, but map everything to accounting categories under the hood.

- school_stage

REVENUE ENGINE RULES
- Revenue lines should be optional and toggleable
- Do not force every user to complete every revenue line
- Each revenue line should use one of these driver types:
  - annual fixed amount
  - monthly amount
  - per student
  - percent of another selected base, only where useful
- Keep the default UI simple by showing only the most relevant lines based on funding profile
- Allow users to add additional revenue lines from the standard list

FUNDING PROFILE DEFAULTS
For tuition_based:
- show tuition and fees first
- show scholarships / discounts
- keep public funding collapsed but available

For charter_public_funded:
- show public funding lines first
- keep tuition visible only if user enables it
- surface grants and food reimbursement more prominently

For hybrid_mixed:
- show tuition, public funding, ESA/voucher, and contributions together

STAFFING MODEL CHANGES
Replace the simple staffing inputs with a staffing table.

Each staffing row should include:
- role name
- function category
- employment type
- FTE
- annualized rate for 1.0 FTE OR annual contract amount
- benefits eligible yes/no
- benefits rate %
- payroll tax rate %
- notes optional

FUNCTION CATEGORIES
- instructional
- school_leadership
- student_support
- operations
- administrative
- other

EMPLOYMENT TYPES
- full_time
- part_time
- contract

STAFFING RULES
- Part-time staff should be modeled via FTE less than 1.0
- Example: 0.5 FTE teacher or 0.25 FTE counselor
- Contract workers should not flow into payroll wages if they are true contractors
- Contract workers should flow to professional services / vendor expense unless explicitly marked as payroll-like
- Allow prebuilt default rows based on school stage and school type, but keep them editable

PERSONNEL COST CALCULATIONS
For employee rows:
- salary expense = FTE × annualized rate
- benefits expense = salary expense × benefits rate, if benefits eligible
- payroll taxes = salary expense × payroll tax rate

For contract rows:
- contract expense = annual contract amount
- no benefits unless user explicitly adds them

Summarize personnel into:
- salaries and wages
- benefits
- payroll taxes
- contracted personnel

IMPORTANT ACCOUNTING RULE
Keep operating expenses on the P&L.
Keep capital and financing items separate from operating expenses.
Do not force a full balance sheet in v1, but do keep capital and debt uses visible in the export and consultant analysis.

EXPENSE DRIVER TYPES

- annual fixed amount
- monthly amount
- per student
- percent of revenue

The UI should default to the most common driver for each line item.

Examples:
- rent = monthly
- utilities = annual fixed
- curriculum = per student
- merchant fees = percent of selected revenue base
- software subscriptions = annual fixed

CONSULTANT ENGINE UPDATE
Make the consultant school-type-aware and accounting-aware.

The consultant should analyze:
- revenue mix
- staffing burden
- occupancy burden
- program spend
- administrative overhead
- capital needs
- debt burden, if entered
- prior-year vs current-year movement for operating schools

Examples of school-type-aware guidance:
- for tuition-based schools, comment on discount rate, tuition dependence, collection risk, and enrollment sensitivity
- for charter/public-funded schools, comment on concentration of public funding, enrollment count dependency, timing risk, and staffing/facility alignment
- for hybrid schools, comment on complexity and concentration across revenue streams

Examples of accounting-aware guidance:
- occupancy costs too high relative to revenue
- contracted personnel unusually high
- curriculum and program costs out of line with enrollment
- software/subscription spend fragmented
- travel and memberships high relative to stage
- founder compensation unrealistically low
- debt service pressure may affect loan readiness
