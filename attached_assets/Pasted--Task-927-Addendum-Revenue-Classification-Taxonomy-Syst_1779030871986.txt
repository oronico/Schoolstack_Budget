## Pinned decisions

### Choose Option 1: reclassify, do not add a Risk Framing note

The base spec offers two paths. Option 1 (reclassify to `contracted_pending_enrollment`) is the correct choice and the task should commit to it before kickoff. Reasoning:

- Option 2 (Risk Framing note) treats the symptom by adding explanation around a misleading number. The number is still misleading; the founder now has a paragraph of caveats.
- Option 1 fixes the taxonomy so the number itself becomes accurate. The packet doesn't need a paragraph to explain why `0.00×` doesn't mean what it appears to mean — the figure reads correctly on its own.
- Lender-grade output benefits from clean primary metrics, not from metrics-plus-asterisks.

Option 1 is more work but lower long-term cost. Take it.

### Revenue classification taxonomy

Replace the current binary contracted-vs-policy-dependent split with a four-bucket taxonomy:

```typescript
type RevenueClassification = 
  | 'contracted_signed'         // signed contract with named counterparty, e.g., 
                                // multi-year service contract, signed enrollment 
                                // agreements

  | 'contracted_pending_enrollment'  // per-pupil payment structurally owed once 
                                     // students are enrolled and verified eligible 
                                     // (ADM, voucher per-pupil rates)

  | 'projected_enrollment_driven'    // tuition revenue from enrolled students; 
                                     // depends on collection rate and retention 
                                     // but is otherwise predictable

  | 'discretionary'             // grants, donations, fundraising — depend on 
                                // continued funder relationships and discretion
```

The current `policy_dependent` bucket dissolves into either `contracted_pending_enrollment` (if the payment is owed per-pupil under existing statute) or `discretionary` (if the appropriation itself depends on annual legislative action).

Each bucket carries different weight in Hard Revenue Coverage:

```
hard_revenue = sum(contracted_signed) + 
               sum(contracted_pending_enrollment) × enrollment_certainty_factor
```

Where `enrollment_certainty_factor` is a documented multiplier (e.g., 0.85 for Y1 of a startup, 0.95 for operating schools with retention history). This factor:
- Acknowledges enrollment risk explicitly
- Is calibrated, not invented per render
- Lives in the same canonical location as the taxonomy

`projected_enrollment_driven` and `discretionary` revenue are excluded from Hard Revenue Coverage entirely (as they are today).

### Storage location and citation discipline

The taxonomy lives in a dedicated module:

```
lib/finance/src/classification/revenue-classification.ts
```

Each bucket carries:
- `bucketId`: enum value
- `description`: human-readable definition
- `examples`: list of concrete examples (e.g., "FL FES-EO voucher", "AZ ADM per-pupil")
- `inclusionInHardRevenue`: boolean or weighted multiplier
- `rationale`: paragraph explaining the lender-relevant reasoning
- `lastValidated`: YYYY-MM-DD
- `notes`: edge cases, jurisdictional variations

The `RevenueRowData` schema gets a `classification` field of type `RevenueClassification` and a `classificationRationale` field for the per-row reasoning ("AZ ADM per-pupil — contracted pending enrollment per ARS §15-185").

### Per-row classification migration

Existing seed data needs migration. Apply these rules to the three persona fixtures and document any judgment calls:

**Riverside Christian Academy (FL private + FES-EO):**
- Annual Tuition (K-8): `projected_enrollment_driven`
- Registration & Activity Fees: `projected_enrollment_driven`
- Scholarships / Financial Aid: discount, not revenue — no classification change
- FL FES-EO Voucher: `contracted_pending_enrollment` (per-pupil rate is statutorily defined)
- Annual Fund Donations: `discretionary`
- Foundation Grants: `discretionary`
- After-School Programs: `projected_enrollment_driven`

**Liberty STEM Charter (AZ charter):**
- State Per-Pupil Funding (K-5, 6-8, 9-12): `contracted_pending_enrollment` (AZ Basic State Aid is statutorily defined)
- Title I Federal Funding: `contracted_pending_enrollment` if formula-based, `discretionary` if competitive grant
- Charter Startup Grant (CSPP): `discretionary` (competitive grant, not formula)
- After-School & Summer Programs: `projected_enrollment_driven`

**Oakwood Learning Studio (AZ microschool tuition-based):**
- Private Pay / Tuition: `projected_enrollment_driven`
- Registration / Enrollment Fees: `projected_enrollment_driven`
- Scholarships / Financial Aid: discount, not revenue
- Donations / Fundraising: `discretionary`
- Startup Grant (Foundation): `discretionary`

Document the per-row classification decisions in code comments at the seed-data definition site and in the completion summary.

### Backward compatibility

Existing founder models in production carry the old binary classification. Two paths:

1. **Migration script:** Map old `policy_dependent` to new buckets per row. Run once, manually verify the result, persist.
2. **Inference layer:** When loading a model with the old shape, infer the new classification from row metadata (source, line item type). Migrate on next save.

Option 2 is safer for early-stage data but creates inference logic that itself needs maintenance. Option 1 is cleaner but requires a careful migration. The agent should pick based on how much production data exists; if it's a handful of seed fixtures plus test data, option 1 is fine.

### Enrollment certainty factor

Don't hard-code `0.85` or `0.95`. Store it in the classification module with rationale:

```typescript
export const ENROLLMENT_CERTAINTY_FACTORS = {
  startup_year_1: {
    factor: 0.85,
    rationale: 'Year 1 enrollment for startup schools carries highest 
      uncertainty; 15% haircut reflects typical first-year attrition 
      and over-projection',
    source: '[citation pending]',
    lastValidated: '2026-05-17'
  },
  operating_with_retention_history: {
    factor: 0.95,
    rationale: 'Operating schools with at least 2 years of retention 
      data carry lower enrollment risk',
    source: '[citation pending]',
    lastValidated: '2026-05-17'
  }
  // additional factors as needed
};
```

Same citation discipline as #926. `[citation pending]` is acknowledged in the completion summary and covered by the architectural follow-up.

## Prevention mechanism

Once #927 lands, the following prevent the original coarseness from returning:

1. **Type-level taxonomy:** `RevenueClassification` is a TypeScript union type with four values. Adding a new bucket is a type change; misclassifying a row is a compile-time error if the seed data uses the literal types.

2. **Required per-row classification:** `RevenueRowData.classification` is required, not optional. A row without classification is a schema validation error at write time.

3. **Single computation path:** `computeHardRevenue` is the only function that computes Hard Revenue. Consumers reference its output, never reimplement. A second implementation is caught by code review (and ideally by a unit test that asserts the function is referenced from exactly one location).

4. **Citation discipline at type level:** Enrollment certainty factors require `source` and `lastValidated` fields. An unsourced factor is a compile-time error.
