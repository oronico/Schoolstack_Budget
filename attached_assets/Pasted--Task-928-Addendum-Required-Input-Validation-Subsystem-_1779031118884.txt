**Apply to:** Task #928 — Require Tuition Collection Rate input
**Base task spec:** Use the existing #928 spec as-is, with the additions below
**Effective:** Before kickoff

**Framing reminder:** This task improves the required-input validation subsystem. Liberty's `Tuition collection rate: Not entered` is evidence the subsystem accepts silent defaults on critical inputs — not the deliverable. ## Pinned decisions

### Conditional requirement, not blanket

Tuition Collection Rate is required for **tuition-based or hybrid funding profiles**. For pure public-funded charters (no tuition revenue), the field is not applicable. The validation rule:

```
required = (fundingProfile in ['tuition_based', 'hybrid']) 
        OR (any revenue row has classification 'projected_enrollment_driven' 
            AND source contains 'tuition')
```

The second clause catches schools that label themselves "public_funded" but have a residual tuition revenue stream (private-pay students at a charter, summer programs, etc.). Don't rely on `fundingProfile` alone — the row-level signal is more reliable.

For Liberty STEM (pure charter, no tuition rows): field is not applicable. The wizard should skip it gracefully, not show a required-field error.

### Validation taxonomy

Introduce a typed validation state for assumption fields:

```typescript
type AssumptionRequirement =
  | { state: 'required'; reason: string }
  | { state: 'required_conditional'; condition: ConditionDescriptor; reason: string }
  | { state: 'optional' }
  | { state: 'deprecated'; replacedBy?: string };

type ConditionDescriptor = {
  predicate: (model: ModelState) => boolean;
  description: string;  // human-readable, surfaced in wizard tooltips
};
```

Each assumption in `assumption-registry.ts` carries an `AssumptionRequirement`. Tuition Collection Rate becomes:

```typescript
{
  state: 'required_conditional',
  condition: {
    predicate: (m) => isTuitionBearingModel(m),
    description: 'Required for tuition-based or hybrid funding profiles, 
      or any model with tuition-derived revenue rows'
  },
  reason: 'Drives collection-rate-adjusted revenue calculation; missing 
    value produces overstated revenue and understated cash risk'
}
```

The `reason` field surfaces in the wizard tooltip AND in the server error response when packet generation is rejected — so founders see WHY the field is required, not just THAT it is.

### Three-layer enforcement

Enforcement runs at three layers, in order of strictness:

1. **Wizard UI (soft):** inline validation error on the field; required-step indicator on the step navigator; CTA on the wizard summary screen pointing at the missing field. Founders can navigate away from the step but cannot mark it complete.

2. **Schema validation (strict):** Zod refinement on the wizard submission payload. Submitting a payload without the required field returns a structured validation error pointing at the specific field.

3. **Server route (strict):** packet generation endpoint validates against the same Zod schema. Even if a client bypasses the wizard UI, the server returns a 422 with the field-level error. The wizard maps this back to a "complete this step" CTA.

All three layers reference the SAME source-of-truth — the `AssumptionRequirement` on `assumption-registry.ts`. No layer hand-codes its own required-field list.

### Server error shape

The server response when required fields are missing:

```typescript
{
  error: 'incomplete_model',
  missingRequiredFields: [
    {
      fieldId: 'tuitionCollectionRate',
      reason: 'Drives collection-rate-adjusted revenue calculation...',
      conditionDescription: 'Required for tuition-based or hybrid 
        funding profiles...',
      wizardStep: 'revenue',
      wizardFieldPath: 'tuition.collectionRate'
    }
  ]
}
```

The wizard consumes this shape to render the CTA accurately. The `wizardStep` and `wizardFieldPath` mean the CTA can deep-link to the missing field, not just suggest a step.

### Backward compatibility

Existing founder models may have null tuition collection rates that were silently defaulted. Two paths:

1. **Surface the gap on next open:** When a founder opens an existing model with a missing required field, the wizard shows the required-step indicator and routes them to the missing field on next interaction. Generation continues to work for models that already generated successfully (grandfathered), but re-generation requires completion.

2. **Block all generation:** Existing models with missing fields cannot generate at all, requiring backfill before next packet.

Option 1 is the right call for early-stage product — don't punish founders for the system's prior permissiveness. Document the grandfathering rule in code comments at the validation enforcement point.

## Prevention mechanism

Once #928 lands, the following prevent silent-default recurrence at the class level:

1. **Typed validation state:** `AssumptionRequirement` is a discriminated union. Adding a new field to the assumption registry requires explicitly choosing a validation state at the type level. Forgetting to specify is a compile-time error.

2. **Single source of truth:** Wizard schema, server schema, and engine all derive from `assumption-registry`. A new required field added to the registry automatically propagates to all three layers. No hand-coded field lists.

3. **Required reason field:** Every `required` and `required_conditional` entry must include a `reason` string. An unsourced requirement is a compile-time error.

4. **Linting rule (optional, file as follow-up):** Warn on engine code that uses `||`, `??`, or explicit default fallbacks on values that come from required assumption fields. The pattern `field || 1.0` is suspicious; it suggests the engine is masking a missing-required-field case.
