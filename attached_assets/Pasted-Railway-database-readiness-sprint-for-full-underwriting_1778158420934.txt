Railway database readiness sprint for full underwriting.

Repo:
oronico/Schoolstack_Budget

Goal:
Confirm the Railway Postgres database supports the current public guest wizard and authenticated budget model today, then design the schema extension required for full underwriting. Do not break existing users, saved models, exports, auth, shared links, or analytics.

Current DB state:
The app already has tables for users, schools, financial_models, exports, events, rate_limits, feedback, error_logs, shared_links, and pending_signups. The financial_models table stores most model data in JSONB. That is acceptable for budgeting, but not sufficient for full underwriting.

Part A. Verify Railway production database

1. Confirm Railway has a Postgres service attached.
2. Confirm API service has DATABASE_URL set.
3. Confirm /api/ready returns 200 and db: connected.
4. Confirm schema tables exist:
   - users
   - schools
   - financial_models
   - exports
   - events
   - feedback
   - error_logs
   - rate_limit_hits
   - shared_links
   - pending_signups
5. Confirm indexes exist for:
   - financial_models_user_id_idx
   - schools_user_id_idx
   - exports_user_id_idx
   - exports_model_id_idx
   - events_user_id_idx
   - events_event_name_idx
6. Confirm migrations or schema push process works in production.
7. Confirm Railway backups are enabled.
8. Confirm no public endpoint can expose raw database contents.
9. Confirm error_logs.request_body is sanitized and does not store passwords, tokens, secrets, bank account numbers, or PII beyond what is necessary.
10. Confirm data retention expectations are documented.

Return a short Railway DB health report:
- connected: yes/no
- tables present
- indexes present
- backups enabled
- migration strategy
- risks

Part C. Implement only safe foundation if approved

After schema plan is reviewed, implement Phase 1 tables only:
- underwriting_applications
- underwriting_documents
- underwriting_evidence
- underwriting_metrics_snapshots
- eligibility_gate_results
- audit_log

Do not yet build UI for everything.
Do not migrate existing financial_models into applications automatically.
Do not delete existing JSONB fields.

Part D. Migration requirements

1. Use Drizzle schema files under lib/db/src/schema.
2. Export new tables through lib/db/src/schema/index.ts.
3. Generate migration using:
   pnpm --filter @workspace/db run generate
4. Apply locally or staging using:
   pnpm --filter @workspace/db run migrate
5. Do not use push-force in production.
6. Add indexes:
   - application_id on all child tables
   - user_id on underwriting_applications
   - status on underwriting_applications
   - document_type on underwriting_documents
   - verification_status on underwriting_documents
   - gate_code on eligibility_gate_results
   - entity_type/entity_id on audit_log
7. Add foreign keys with sensible delete behavior.
8. Avoid storing raw full EINs, bank account numbers, SSNs, or sensitive financial credentials.
9. Document all PII decisions.

Part E. Tests

Add DB/schema tests or smoke scripts that verify:
- application can be created
- document can be attached to application
- evidence can be linked to a document
- metrics snapshot can be saved
- eligibility gate result can be saved
- audit log writes on update
- deleting a user cascades or preserves records according to policy
- no migration breaks existing financial_models
