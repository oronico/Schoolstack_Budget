## Pinned decisions

### Benchmark data shape and storage location

```typescript
type PerPupilBenchmark = {
  state: string;                    // ISO state code: 'AZ', 'FL', etc.
  schoolType: SchoolType;           // 'charter' | 'private' | 'microschool' | etc.
  fundingProfile?: FundingProfile;  // optional refinement: 'tuition_based' | 'voucher_eligible' | 'public_funded'
  typicalLow: number;               // bottom of typical range, in USD/pupil
  typicalHigh: number;              // top of typical range
  ceiling: number;                  // above this triggers the warning
  source: string;                   // human-readable citation (e.g., "AZ DOE Equalization Assistance Schedule 2024-25")
  sourceUrl?: string;               // permalink if available
  lastValidated: string;            // YYYY-MM-DD
  notes?: string;                   // exceptions, weighting context, SPED considerations
};
```

Store the benchmark table in a dedicated module:

```
lib/finance/src/benchmarks/per-pupil-revenue.ts
```

Not inline in `revenue-quality.ts`. Benchmarks are reference data with citation and validation-date metadata; they need their own home so future calibration tasks know where to look.

### Warning severity and lifecycle

The warning is **soft validation**, not a render-prevention. Specifically:

- Warning appears in the Flagged Assumptions section of the lender packet
- Founder can acknowledge / dismiss the warning in the wizard
- Acknowledgment is persisted on the model record with timestamp and acknowledger
- Acknowledged warnings still render in the lender packet, but with an "acknowledged by founder" suffix
- Unacknowledged warnings render with full severity language

This matters because lenders should see whether the founder noticed and accepted the warning, not just that the warning was triggered. The acknowledgment trail is part of the assumption-confidence story.

### Lookup behavior when no benchmark matches

If the engine computes per-pupil revenue for a `(state, schoolType, fundingProfile)` combination that has no benchmark entry:

- Do NOT emit a warning
- Do NOT silently pass through
- DO emit a different flag: "No per-pupil benchmark available for {state} {schoolType}. Verify rate against state appropriation schedule."

This flag has informational severity (lower than the over-ceiling warning) and tells founders the system isn't validating their input. Honest about coverage gaps is better than silent gaps.

## Prevention mechanism

Once #926 lands, the following prevent recurrence at the class level:

1. **Centralized benchmark module:** All per-pupil and per-student benchmarks live in one file with citation metadata. Future contributors adding a benchmark know where to put it. Future readers can audit citations centrally.

2. **Standard flag emission pattern:** The `evaluatePerPupilRevenue` function returns a typed result; the flag emission code uses a typed pattern. Future benchmark types (personnel %, facility cost per student) use the same shape.

3. **Test architecture supports new fixtures:** Adding a new persona to the test suite triggers all benchmark assertions without code changes. New benchmarks added to the module trigger validation for any matching persona automatically.

4. **Citation discipline:** Every benchmark entry requires `source` and `lastValidated` fields at the type level. A benchmark without citation is a compile-time error. This prevents the table from drifting into stale or unsourced data without anyone noticing.
