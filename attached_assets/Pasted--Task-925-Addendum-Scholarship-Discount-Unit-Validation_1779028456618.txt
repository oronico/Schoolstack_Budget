# Task #925 Addendum — Scholarship / Discount Unit Validation System

**Apply to:** Task #925 — Validate scholarship/discount as percentage, not raw value
**Base task spec:** Use the existing #925 spec as-is, with the additions below
**Effective:** Before kickoff

**Framing reminder:** This task improves the input-validation subsystem for unit-bearing financial fields. The Riverside `$12` and Oakwood `$8` renders are symptoms — evidence that the subsystem accepts ambiguous input. The deliverable is a subsystem that makes this class of confusion impossible, not three corrected PDFs.

---

## Pinned decisions

### Scope of the unit-toggle pattern

The unit toggle (PCT vs USD) introduced here is not specific to the scholarship field. Other financial inputs across the wizard likely have the same ambiguity:

- Scholarship / Financial Aid (this task)
- Discount Rate (this task)
- Collection rate adjustments
- Fee waivers
- Annual fund / grant flow rates

Apply the unit-toggle pattern as a **reusable input component** (e.g., `UnitInput`), not a one-off control on `RevenueStep.tsx`. The sibling bug search step (below) identifies other fields that should adopt the pattern.

If full rollout to all candidate fields is larger than #925's scope, file the rollout as a follow-up. But the component itself must be reusable, not field-specific.

### Storage shape

Adopt this canonical storage shape for any unit-bearing financial field:

```typescript
type UnitBearingValue = {
  value: number;        // raw numeric value as entered
  unit: 'PCT' | 'USD';  // explicit, no default
  // Optional: enteredAs preserves the original string for audit
  enteredAs?: string;
};
```

A field stored as `-8` with no `unit` is invalid going forward. Migration of existing data is out of scope per the base spec, but the schema must reject ambiguous values on any new write.

### Render contract

The renderer reads `unit` and formats accordingly:

```
unit === 'PCT' && value < 0  →  "(8.0%)"
unit === 'PCT' && value > 0  →  "8.0%"
unit === 'USD' && value < 0  →  "$(8,000)" or "-$8,000" (pick one, document)
unit === 'USD' && value > 0  →  "$8,000"
unit missing                 →  render error, do not silently format
```

The "render error" case is critical. Today the renderer silently prefixes `$` to whatever is stored. Going forward, missing unit data should produce a visible error in the rendered output (e.g., `[unit missing — field needs revision]`), not a silently wrong number. A lender seeing `[unit missing]` is recoverable; a lender seeing `$8` and trusting it is not.

### Negative-USD convention

Pick one and document in the renderer helper:

- **Accounting convention:** `$(8,000)` for negatives
- **Standard convention:** `-$8,000` for negatives

Recommendation: accounting convention `$(8,000)` for financial discount/scholarship fields. Matches how lenders read financial statements.

## Prevention mechanism

Once #925 lands, the following prevent recurrence at the class level:

1. **Schema-enforced unit field:** Any field typed as `UnitBearingValue` must have both `value` and `unit` at write time. A field stored without `unit` is a schema validation error, not a silent default.

2. **Render contract enforcement:** The render helper does not format a unit-bearing value when unit is missing — output is an explicit error token, never a guessed number.

3. **Reusable UnitInput component:** Any new wizard field accepting a unit-bearing value uses the component. Field-by-field reinvention is prevented by code review.

4. **Lint rule (optional, file as follow-up):** Warn on any new `$\\${value}` template-string usage in the renderer that doesn't go through `formatUnitBearingValue`.
