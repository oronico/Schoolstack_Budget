# Task #929 Addendum — Confidence-Gated Rating Subsystem

**Apply to:** Task #929 — Evidence-tag cap on Lender Readiness
**Base task spec:** Use the existing #929 spec as-is, with the additions below
**Effective:** Before kickoff

**Framing reminder:** This task improves the confidence-gated rating subsystem. The Riverside/Oakwood `Lender Readiness: Strong` at `0% weighted evidence` is evidence the subsystem does not gate ratings on confidence — not the deliverable. The deliverable is a rating subsystem where confidence and underlying-metric quality both gate the final rating, for any current or future persona.

The three persona fixtures test the subsystem. They are not the targets of the fix.

## Architectural lessons from #925, #926, #927, #928

Before starting:

1. **From #925 (unit-bearing fields):** Check whether `confidenceScore` already carries enough structure to drive a cap, or whether new fields need to be introduced. If the existing shape works, extend it. Surface the question if you find a conflict.

2. **From #926 (benchmark validation):** Threshold tables with multiple boundaries (the 25% / 50% cap thresholds here) belong in structured metadata, not magic numbers in the engine. Each threshold deserves a documented rationale.

3. **From #927 (taxonomy):** Rating-cap states are a small typed taxonomy. Prefer a discriminated union over magic strings.

4. **From #928 (validation enforcement):** Confidence-gating is a parallel pattern to required-field validation. Both gate output generation on input quality. Look for shared primitives — the "reason" surfacing pattern from #928 (reason in tooltip AND in error response AND in lender packet) is directly reusable here.

Apply all four lessons. The cap delivered here should be reusable for other ratings that may benefit from confidence gating.

## Pinned decisions

### Rating cap as a structured result, not a string

The rating computation should return a structured object capturing both the uncapped rating and the cap applied:

```typescript
type LenderReadinessResult = {
  uncappedRating: LenderReadinessRating;       // what the metrics alone would say
  effectiveRating: LenderReadinessRating;       // what gets displayed after cap
  cap: {
    applied: boolean;
    capTier?: RatingCapTier;
    reason: string;                              // surfaces in callout
    pendingEvidenceCount: number;                // N of 22 untagged
    totalAssumptionCount: number;
  };
};
```

This structure flows through the engine to the packet builder to the renderer. Consumers display whichever fields they need (the in-app card might show both the uncapped and effective rating with the cap explanation; the lender packet might show only the effective rating with a callout). No consumer recomputes the cap.

### Callout copy template

The cap callout in both surfaces uses a canonical template:

```
"Rating capped at {effectiveRating} pending evidence tagging on 
 {pendingEvidenceCount} of {totalAssumptionCount} assumptions. 
 {capTier.rationale}"
```

Storing the copy template centrally (in the same module as the cap tiers) means future copy changes happen in one place. Don't hand-write the callout in `lender-packet-pdf.ts` or `LenderReadinessCard.tsx`.

### "Strong" floor for lender-grade output

Apply a related rule that the base spec doesn't explicitly call out but is implied:

> The `effectiveRating` value `'Strong'` is the highest tier. It can only be reached when (a) underlying metrics support it AND (b) `taggedFraction >= 0.50`.

This prevents a 100%-tagged model with weak metrics from being labeled "Strong" — the underlying-metric assessment still applies. The cap subsystem produces a ceiling, not a floor.

### Bidirectional consistency requirement

The in-app `LenderReadinessCard` and the lender packet PDF must always render the same `effectiveRating` for the same model state. The components share a data source (the `LenderReadinessResult` object); they should also share rendering primitives where practical. The regression test should assert this consistency explicitly.

## Prevention mechanism

Once #929 lands, the following prevent the original "overconfident rating despite no evidence" pattern from returning:

1. **Typed cap tier table:** `RatingCapTier` carries threshold boundaries and rationale at the type level. Adding a new tier or changing thresholds is a single edit in one location; consumers automatically reflect it.

2. **Structured rating result:** `LenderReadinessResult` carries both uncapped and effective ratings explicitly. Consumers reading only `effectiveRating` cannot accidentally claim the underlying metrics are weak — the cap shape makes the distinction explicit.

3. **Single computation path:** `computeLenderReadiness` is the only function that produces the rating result. No surface recomputes; consumers display from the shared object.

4. **Cross-surface consistency test:** A unit test asserts that any rendering of `effectiveRating` in the packet matches the rendering in the in-app card for the same model state. Catches divergence at the test layer.

5. **Citation discipline at type level:** Each `RatingCapTier` requires `source` and `lastValidated`. Threshold changes without citation are a compile-time error.
