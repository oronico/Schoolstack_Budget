The container starts, runs migrations successfully, then immediately crashes on a Node.js TypeError before the application can serve any requests. It's restarting repeatedly — the log shows the same crash sequence five or six times.
What the error actually says:
TypeError: The "path" argument must be of type string or an instance of URL. Received undefined
    at fileURLToPath (node:internal/url:1605:11)
    at Object.<anonymous> (/app/index.cjs:1211:52297)
fileURLToPath() is a Node.js built-in that converts a file:// URL to a filesystem path. It's getting called with undefined. The most common cause: code that uses import.meta.url (an ES modules feature) got bundled to CommonJS, where import.meta.url is undefined. The bundler produced /app/index.cjs (note the .cjs extension — CommonJS), and somewhere in that bundle, code that worked in ESM is now passing undefined to fileURLToPath.
This is a build/bundling configuration issue, not a runtime application issue. The application code never gets to execute.
Where this typically comes from:
Common patterns that produce this error after a bundler emits CJS:
javascript// Pattern 1 — direct
const __filename = fileURLToPath(import.meta.url);

// Pattern 2 — for __dirname equivalent in ESM
const __dirname = path.dirname(fileURLToPath(import.meta.url));

// Pattern 3 — resolving paths relative to the current module
const configPath = fileURLToPath(new URL('./config.json', import.meta.url));
In ESM these work. After bundling to CJS, import.meta.url is undefined, the URL constructor or fileURLToPath receives undefined, and the process crashes at module load time — before any request handler runs.
Almost certainly related to recent work:
Per the active engagement, recent tasks have introduced new file-based artifacts: migration plan generators, integrity report writers, baseline JSON files for M4, fixture auto-discovery, the math-integrity report at artifacts/api-server/reports/. Any of these that uses import.meta.url to locate its own directory or read sibling files would crash the bundle in production CJS mode.
The fact that the crash is in /app/index.cjs at a specific offset (1211:52297) means the bundler concatenated the offending module into the main bundle. A source-map lookup or a search for fileURLToPath in the api-server codebase will find the culprit in minutes.
