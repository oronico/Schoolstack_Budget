## Pinned decisions

### Approach: rename for uniqueness, not deduplicate

The two delay scenarios encode different lender-facing risk questions and the subsystem should keep both. Removing one collapses two distinct analytical views into one.

Rename to:
- `Revenue Delayed 3 Months — full revenue stack`
- `Public Funding Delayed 3 Months — public sources only`

Document the input transformation each scenario applies in code comments at the scenario definition site.

### "Toughest stress" criterion

When the Lender Commentary generator names "the toughest stress," the deterministic criterion is:

> Lowest Min DSCR across all stress scenarios for the given persona.

If two scenarios tie on Min DSCR, secondary sort is largest Y1 net income decline vs base. Document this criterion in the Commentary generator helper.

### Sequencing with #913

Task #913 (stress-test aggregator into Lender Commentary) consumes the labels this task changes. Two paths:

- **If #913 has not merged:** Sequence #924 first. #913's regression test uses the post-rename labels.
- **If #913 has merged:** Update #913's regression test to use post-rename labels as part of #924's scope.

Confirm #913's status before kickoff.

## Prevention mechanism

Once #924 lands, add the following to the subsystem to prevent recurrence at the class level:

1. **Type-level uniqueness:** Define a `StressScenarioId` enum or union type. Every scenario references an ID, not a string label. Display labels are derived from IDs. A duplicate ID is a compile-time error.

2. **Single-result contract:** Each scenario ID maps to exactly one computation function. Consumers (table, Commentary) reference scenarios by ID and receive the canonical result. No consumer recomputes.

3. **Unit test on the scenario registry:** A test that loads all scenario definitions and asserts:
   - All IDs are unique
   - Every ID has exactly one computation function
   - Every consumer's reference to a scenario uses a valid ID

If items 1–2 are larger than #924's scope, file them as an architectural follow-up (see below).
